# Mobile Decision-First PRD v1

## 1. 文档信息
- 文档名称：Mobile Decision-First PRD v1
- 产品名称：予选（MatchUp）
- 当前主链路：`/m` -> `/m/choose` -> `/m/[category]/profile` -> `/m/[category]/result`
- 当前北极星：首访用户从进入主链路到到达结果页的完成率
- 当前阶段目标：先把主链路打穿，再考虑 utility 辅链的承接与回流
- 关联文档：[Mobile Result Intent Routing PRD v1](./mobile-result-intent-routing-prd-v1.md)

## 2. 产品定义
- 予选是个护决策工具，不是百科站、对比站或展示型官网。
- 用户核心需求不是“再看更多信息”，而是“更快得到一个更适合自己的结论”。
- 产品核心承诺不是“功能齐全”，而是“少量问题，直接收敛到结果”。
- `wiki`、`compare`、`me` 保留，但它们是辅链，不是首访叙事中心。
- 全场景内容继续保留在供给层，但前台必须表现为一个窄、快、稳的决策产品。

## 3. 背景与问题
- 旧叙事把首页、品类入口、utility 能力和后台能力混在一起讲，用户第一秒不知道该点哪里。
- 首页一旦开始承担产品地图教学，就会把最贵的首访流量浪费在解释自己，而不是推动用户进入决策。
- `/m/choose` 如果继续承担“解释产品价值”的职责，就会从一个分发页变成第二个首页，转化一定发胖。
- 分析口径如果继续从 utility 链路开始讲，就会掩盖真正的主链路问题。

## 4. 目标
- 让新用户在 3 秒内知道这是一个决策工具，而不是功能合集。
- 让新用户在 1 次会话内更高概率拿到结果页。
- 让首页和 choose 页都只做一件事，降低首访选择负担。
- 用统一叙事约束前端页面、分析口径和后续迭代取舍。

## 5. 非目标
- 不在本期移除 `wiki`、`compare`、`me`
- 不在本期改造 backend / frontend 的部署切割
- 不在本期取消全场景内容供给
- 不在本期做更复杂的 BI 平台或全站分析体系
- 不在本期扩展新的首访一级入口

## 6. 目标用户
- 已经有购买或换新意图，但不想自己查成分、翻评测、做复杂比较的人
- 有明确品类需求，但不确定自己更该优先解决什么问题的人
- 需要快速结论和下一步，而不是需要完整知识库的人

## 7. 体验原则
- 首访只讲主链路
- 每一屏只做一件事
- 先结论，后解释
- utility 能力以下沉和回环为主，不抢首页解释权
- 后台供给能力不直接外溢成前台复杂度

## 8. 北极星与支撑指标

### 8.1 北极星
- 首访用户从进入主链路到到达结果页的完成率

### 8.2 支撑指标
- `/m` 主 CTA 点击率
- `/m/choose` 到答题开始率
- 答题完成率
- 结果页 CTA 点击率

### 8.3 当前降级指标
- `wiki` CTR
- `compare` 阶段细分
- `feedback` 分布
- 环境切片

## 9. 信息架构

### 9.1 `/m`
页面目标：
- 让用户立刻理解产品价值
- 让用户立刻点击主 CTA

页面必须包含：
- 一句主标题
- 一句副标题
- 一个主 CTA：`开始测配`
- 一个弱次入口：`继续上次进度`
- 一条可信理由说明

页面不得包含：
- 四入口教学
- 新手导览流程
- 产品地图式介绍
- 把 `wiki`、`compare`、`me` 做成同级首访决策入口

推荐文案骨架：
- Eyebrow：`个护决策工具`
- H1：`更快选到适合自己的护理方案`
- Body：`回答少量问题，直接得到更适合你的护理方向和产品结果，不用自己逐个比成分。`
- 主 CTA：`开始测配`
- 次 CTA：`继续上次进度`
- 可信理由：`约 30-45 秒｜直接给结果和理由｜少靠猜，少踩坑`

### 9.2 `/m/choose`
页面目标：
- 帮用户在最短时间内选择当前品类
- 帮回访用户恢复上次进度

页面必须包含：
- 页面标题
- 进度恢复卡片
- 五个品类卡片

每个品类卡片只保留：
- 品类名
- 一句适用场景
- 预计时长
- 开始按钮

页面不得包含：
- 重复讲产品价值
- 长篇解释
- 复杂轮播和装饰性交互
- 把 utility 页面变成选择入口

推荐文案骨架：
- 标题：`先选你现在要解决的那一类`
- 副标题：`每类只问少量问题，直接给更合适的方向和结果。`
- 恢复卡片：`你上次做到 {category} 第 {step}/{total} 步，继续即可。`

品类文案建议：
- 洗发水：`头皮出油、头屑、敏感或发丝状态问题`
- 沐浴露：`洗后干涩、长痘、粗糙或想洗得更舒服`
- 护发素：`毛躁、打结、受损或一用就塌`
- 润肤霜：`干痒粗糙、修护续航或质地负担问题`
- 洗面奶：`清洁力、敏感度和洗后肤感拿不准`

## 10. Utility 辅链定位
- `wiki`：补充阅读和结果后续说明
- `compare`：横向比较能力
- `me`：历史、在用、恢复与回访入口

产品约束：
- utility 必须承接主链路，而不是替代主链路
- utility 可以成为结果页之后的下一步，不应成为首访前置教育

## 11. 分析口径
- 分析首页默认从 `/m` 主 CTA 开始，不从 `wiki` 或 `compare` 开始讲故事
- 每周先回答四个问题：
  1. 有多少新用户进入了主链路
  2. 多少人进入了问答
  3. 多少人拿到了结果
  4. 结果后多少人继续动作

## 12. 验收标准
- 新用户进入 `/m` 后，3 秒内知道要点 `开始测配`
- 新用户进入 `/m/choose` 后，3 秒内完成品类选择
- 首页与 choose 页都不再承担产品地图教学
- README、PRD、分析口径不再出现旧的多入口主叙事
- utility 页面继续存在，但其文档定位全部降为辅链

## 13. 实施优先级

### P0
- 收敛首页叙事
- 收敛 choose 页职责
- 统一 README 与 PRD 口径
- 调整 analytics 说明为主链路优先

### P1
- 基于新口径继续压缩首页和 choose 页 UI
- 补足与北极星直接相关的埋点与报表

### P2
- 再看 utility 辅链如何回流
- 再看更细的 compare / feedback / 环境切片分析
